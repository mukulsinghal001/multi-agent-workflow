# Contributing to Multi-Agent Framework Demo

Thank you for your interest in contributing to the Multi-Agent Framework Demo! We welcome contributions from the community to help improve this educational resource.

## üéØ Contribution Guidelines

### Code of Conduct
By participating in this project, you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md). Please be respectful and considerate of others.

### Types of Contributions
We welcome several types of contributions:

1. **Bug Reports**: Help us identify issues
2. **Feature Requests**: Suggest new demo capabilities
3. **Documentation Improvements**: Enhance clarity and completeness
4. **Code Contributions**: Add new features or fix bugs
5. **Example Scenarios**: Add new generic business use cases
6. **Educational Content**: Tutorials, guides, and examples

### Getting Started

1. **Fork the repository** on GitHub
2. **Clone your fork** locally
3. **Create a feature branch**: `git checkout -b feature/your-feature-name`
4. **Make your changes** following our guidelines
5. **Test thoroughly** with demo data
6. **Submit a pull request** with a clear description

## üìù Development Guidelines

### Data Privacy Requirements
- **ALL contributions must maintain data privacy**
- **NO real company names, brands, or proprietary information**
- **USE only generic, hypothetical scenarios**
- **ENSURE all content is safe for public sharing**

### Code Style
- Follow existing code patterns and naming conventions
- Use clear, descriptive variable and function names
- Include docstrings for all functions and classes
- Follow PEP 8 Python style guide
- Keep lines under 120 characters where possible

### Documentation
- Update documentation for any changes
- Use clear, concise language
- Include examples where helpful
- Maintain consistent formatting
- Use Markdown for all documentation files

### Testing
- Test all changes with demo data
- Verify no real data is exposed
- Check guardrails are enforced
- Test edge cases and error conditions
- Ensure backward compatibility

## üêõ Bug Reports

### How to Report Bugs
1. **Check existing issues** to avoid duplicates
2. **Provide clear reproduction steps**
3. **Include expected vs. actual behavior**
4. **Specify environment details** (Python version, dependencies)
5. **Attach relevant logs or screenshots**

### Bug Report Template
```markdown
**Description**: Clear description of the issue

**Steps to Reproduce**:
1. Step one
2. Step two
3. Step three

**Expected Behavior**: What should happen

**Actual Behavior**: What actually happens

**Environment**:
- OS: [Windows/Linux/Mac]
- Python: [version]
- PySpark: [version]
- Gemini CLI: [version]

**Additional Context**: Any other relevant information
```

## üîß Code Contributions

### Pull Request Process
1. **Fork the repository**
2. **Create a feature branch**
3. **Make your changes**
4. **Test thoroughly**
5. **Update documentation**
6. **Submit pull request**
7. **Address review feedback**

### Pull Request Requirements
- Clear title and description
- Reference related issues
- Include before/after comparisons
- Document breaking changes
- Pass all existing tests
- Maintain data privacy

### Code Review Process
1. **Initial review** by maintainers
2. **Feedback provided** within 7 days
3. **Iterative improvements** as needed
4. **Final approval** and merge
5. **Release in next version**
